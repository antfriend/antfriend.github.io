# RFC Folder Review — 2026-05-08

## Executive Summary

The RFC collection is comprehensive and well-conceived, with 26 files spanning 5 major frameworks (TTN, TTDB, TTE, TTCP, A32). However, there are **organizational inconsistencies**, **incomplete references**, and **structural variations** that reduce clarity and professionalism. Most issues are low-impact formatting/consistency problems; none block functionality.

### 2. HIGH — Organizational Issues

#### 2.1 TTN-RFC-MANIFEST.md — Not an RFC
**File:** `RFCs/TTN-RFC-MANIFEST.md`  
**Issue:** This file is a SHA256 checksum manifest, not an RFC specification. It doesn't fit the RFC documentation pattern and shouldn't be indexed alongside actual RFCs.  
**Impact:** Conceptual misplacement; may confuse readers about what the RFCs directory contains.  
**Recommendation:** Either:
- Move to a separate `.checksums/` folder, OR
- Rename to `MANIFEST.md` (without RFC prefix), OR
- Add a note at top explaining it's metadata, not a specification

#### 2.2 RFC Header Format Inconsistency
**Files Affected:** All RFCs  
**Issue:** Different RFCs use different header formats:

**Type A (minimal):**
```markdown
# TTN-RFC-0001
## Title
### Version 1.0
Status: Stable
```

**Type B (comprehensive):**
```markdown
# TTCP-RFC-0001: Record Rendering
**Version:** 1.0
**Status:** Draft
**RFC Number:** 0001
**Project:** toot-toot-engineering
**Component:** TTDB Content Publishing (TTCP)
**Depends on:** [list of dependencies]
**Author:** antfriend
**Created:** 2026-04-28
```

**Impact:** Inconsistency makes navigation harder; some RFCs have rich metadata, others don't.  
**Recommendation:** Standardize on one format. The Type B format (like TTCP-RFC-0001) is more professional and informative. Adopt it for all RFCs.

#### 2.3 Missing or Inconsistent "Depends On" Declarations
**Files Affected:** Most RFCs in TTN, TTE, A32 families  
**Issue:** Some RFCs declare dependencies (TTCP, A32-Amendment-A), but most don't, even when they clearly depend on others.

Examples of undeclared dependencies:
- `A32-RFC-0002` uses TTDB-RFC-0001 but doesn't declare it
- `A32-RFC-0003` uses TTDB concepts but doesn't declare dependencies
- `TTE-RFC-0002` and `TTE-RFC-0003` are related but don't reference each other

**Impact:** Hard to understand the logical order to read RFCs; new readers may miss critical prerequisites.  
**Recommendation:** Add "Depends on" metadata to every RFC that has dependencies.

#### 2.4 Inconsistent End-of-RFC Markers
**Files Affected:** TTN-RFC-0001, TTE-RFC-0001 have `End RFC-0001` or similar; most others don't.  
**Issue:** Some RFCs end with a marker like `End RFC-0001`, most do not.  
**Impact:** Minor; mostly affects readability/clarity.  
**Recommendation:** Standardize: either always use end markers or never use them. (Suggestion: **don't use them** — EOF is sufficient.)

### 3. MEDIUM — Consistency & Naming Issues

#### 3.1 "MyMentalPalaceDB" vs "TTDB" Naming Inconsistency
**Files Affected:** Multiple (TTDB-RFC-0001, A32 RFCs, TTN RFCs, etc.)  
**Issue:** Full name varies:
- `TTDB-RFC-0001` calls it "MyMentalPalaceDB"
- `A32-Index` calls it "MyMentalPalaceDB"
- `companion.md` and `companion_README.md` now call it "Toot-Toot Database" (recently updated)
- TTN-RFC-0001 just says "TTDB"

**Impact:** Confusing for readers; the full name should be consistent.  
**Recommendation:** Choose one canonical name and use it consistently. Suggestion: Since we've re-branded the SAGE companion to LOCUS and rooted it in the Locus framework, consider updating TTDB references to clarify it's "Toot-Toot Database (TTDB)" when the full name is needed, and just "TTDB" elsewhere.

#### 3.2 Status Field Inconsistency
**Files Affected:** All RFCs  
**Issue:** Different status values used:
- "Stable" (TTN-RFC-0001, TTN-RFC-0005, A32-RFC-0002-Amendment-A, TTDB-RFC-0005)
- "Draft" (A32-RFC-0001, A32-RFC-0002, TTCP-RFC-0001)
- "Informational" (TTDB-RFC-0006)

Some have no status field at all.  
**Impact:** Unclear which RFCs are production-ready vs. experimental.  
**Recommendation:** Standardize status values:
  - `Informational` — Framework foundations, non-normative
  - `Proposed` — Candidate for adoption; feedback welcome
  - `Draft` — In progress; expect changes
  - `Stable` — Production-ready; breaking changes rare
  - `Deprecated` — Superseded; use alternative

### 4. MEDIUM — Structural Issues

#### 4.1 Very Short RFCs May Be Incomplete
**Files Affected:**
- `TTE-RFC-0001-Workflow-and-Roles.md` (40 lines)
- `TTE-RFC-0003-Definition-of-Done-and-Release.md` (30 lines)
- `TTN-RFC-0005-Trust-and-Reputation.md` (35 lines)

**Issue:** These RFCs lack depth. For example, TTE-RFC-0001 lists 10 roles but doesn't define what each does. TTN-RFC-0005 outlines a trust model but provides no specifics.  
**Impact:** Readers cannot implement based on these specs alone; they require external context.  
**Recommendation:** Either:
  - Expand these RFCs with concrete definitions/examples, OR
  - Retitle them as "Framework Summary" or "Proposed Framework" rather than full RFCs, OR
  - Cross-link to external documentation that provides the details

#### 4.2 A32-Index.md — Not an RFC, But Valuable
**File:** `RFCs/A32-Index.md`  
**Issue:** This is a metadocument (an index), not an RFC specification. It's valuable for navigation but doesn't fit the RFC pattern.  
**Impact:** Minor; the file is useful and well-written.  
**Recommendation:** Keep it, but consider:
  - Renaming to clarify it's an index (e.g., `A32-RFC-INDEX.md`), OR
  - Moving RFC indexes to a separate `INDEXES/` folder alongside the main INDEX.md

#### 4.3 Missing or Incomplete Examples in Normative RFCs
**Files Affected:** TTDB-RFC-0001, TTDB-RFC-0002, TTDB-RFC-0003, TTDB-RFC-0004  
**Issue:** Some RFCs that define file formats (TTDB-RFC-0001-0004) are dense with rules but lack concrete examples.  
**Impact:** Harder for implementers to understand.  
**Recommendation:** Add "Example" sections to each normative spec.

### 5. LOW — Professionalism & Polish

#### 5.1 TTCP-RFC Files — Header Format Better, But Missing Content Structure
**Files Affected:** TTCP-RFC-0001, TTCP-RFC-0002, TTCP-RFC-0003  
**Issue:** These have the best header metadata, but their content structure is less consistent with other RFC families. Also, TTCP-RFC-0002 and TTCP-RFC-0003 both mention "Depends on TTDB-RFC-0001, 0002, 0003, 0005" — should also mention each other if they're related.  
**Recommendation:** Minor polish. Ensure dependencies are bidirectional where appropriate.

#### 5.2 Terminology Consistency Within RFCs
**Files Affected:** TTN-RFC-0001 through TTN-RFC-0006  
**Issue:** TTN RFCs use some undefined terms without linking to definitions:
- "Semantic Event" (mentioned in TTN-RFC-0001 §2, but not fully defined)
- "Typed Edge" (defined in TTDB-RFC-0003, but TTN assumes knowledge of it)
- "TTAI" (mentioned in TTN-RFC-0001 §3, but what is it?)

**Impact:** Reading order matters; you can't read TTN-RFC-0001 standalone.  
**Recommendation:** Add a "Terminology" section to each RFC family that cross-links to definitions in related RFCs, or add inline definitions.

#### 5.3 Links in INDEX.md Are Relative, Not Anchored
**File:** `RFCs/INDEX.md`  
**Issue:** Filenames are listed but not hyperlinked. A reader viewing INDEX.md can't click to open an RFC.  
**Recommendation:** Convert filenames to markdown links:
```markdown
- [TTN-RFC-0001.md](TTN-RFC-0001.md): Core Semantic Mesh Specification
```
